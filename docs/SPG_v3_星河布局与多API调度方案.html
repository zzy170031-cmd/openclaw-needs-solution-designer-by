Galaxy-first Product Surface

SPG 不再只是数据后台，而是一张可导航、可解释、可扩张的星河界面。

页面大结构延续你给的参考稿思路，但中间不再是平铺卡片区，而是换成一个“星系中枢”。它承载 赛道核心、热点风暴、实体资产、来源轨道 和 事件星团，让页面第一眼就像在看一个正在运转的游戏宇宙，而不是一张传统的信息表。

你这次明确的前提
  • 布局风格星河类整体页面
  • 中心区域替换成星系图案
  • 数据接入多 API 混合调度
  • 工程语言Rust

页面骨架

页面不是简单套参考稿，而是在它的框架上重新分配视觉主次。你给的样式里，信息密度很足，但中心区没有足够强的“世界观”；这一版会把中间区域变成真正的星系舞台。

01 Hero

顶部星河舞台

首屏就是“赛道宇宙总览”。中心是银河核心，外圈是来源、事件、实体、提醒节点。它不只是装饰，而是总导航与状态摘要。

  • 核心标的强势能显示在银河内核
  • 外围节点按信号强度和可信度聚类
  • 点击节点可下钻到实体页或事件页
02 Narrative

中段分析带

紧跟在银河舞台之后的，不是普通表格，而是“赛道今日主线”“热点风暴”“来源结构变化”“异常预警”四块横向分析区。

  • 强调今天赛道真正发生了什么
  • 让热度变化带有解释链，而不是只有数字
  • 适合后续接 AI 自动总结
03 Drilldown

底部深度工作区

用于放实体库、事件时间线、未决池、来源治理、抓取状态、配额面板。也就是说，真正“工程控制”的部分会沉到底部，而不是抢首屏视觉。

  • 实体资产与事件分层浏览
  • unresolved_pool 复核位
  • API 调度与缓存命中监控

Motion

动效原则

  • 星系轨道做慢速旋转，让页面“活着”，但不做廉价粒子雨。
  • 热点节点只在 hover 或数据刷新时做轻微脉冲，不连续闪烁。
  • 页面进入动效以“雾化显现 + 节点铺开”为主，不做飞入式乱动。
  • 重点是让用户感觉“系统正在运转”，不是“网页在表演”。

授权与配置入口

API 不应该通过改代码来接入，而应该有一个独立的网页配置入口。系统首次打开时，先进入 授权与配置页，完成 API Key、Base URL、配额、默认角色与连通性验证后，才允许进入正常工作台。配置完成后，这个入口也会一直保留在系统右上角，方便后续调整。

Gateway Rule

进入系统前的放行条件

  • 首次访问默认跳转到 /setup/providers，而不是直接进入主工作台。
  • 必须先完成核心链路配置与验证，才能进入星河主界面。
  • 建议核心放行条件：豆包、Serper、以及至少一个深抓或校验能力已通过测试。
  • 如果后续某个关键 API 失效，系统顶部出现“配置异常”入口，但不需要重新改代码。
UX Entry

页面里必须有的配置入口

  • 首次启动：强制进入 API 授权页。
  • 正常运行后：右上角保留 API Control / Provider Settings 按钮。
  • 每个 Provider 卡片上直接显示状态：未配置 / 已配置 / 测试失败 / 正常。
  • 支持重新测试、暂停调用、调低预算、切换主次角色。
Page Flow

推荐入口流转

首次打开
  -> /setup/providers
  -> 填写 API Key / Base URL / 配额参数
  -> 测试连接
  -> 保存到加密配置存储
  -> 解锁 /galaxy 主工作台

正常运行
  -> /galaxy
  -> 顶部导航保留 “API Control”
  -> 随时进入 /settings/providers 调整供应商状态与预算

多 API 角色分工

你的 API 资源已经足够做一套高可用的混合链路，但前提是不能“见内容就全调”。这版方案默认以 Serper 负责发现、Firecrawl 负责高价值正文抽取、豆包负责日常解析 为主链，Exa / Tavily / Gemini 做精查和回退。

Provider 月配额 默认角色 适合场景 使用原则
Serper 2500 主搜索入口 关键词发现、站点结果页、热点初筛 默认先走它，所有 query 先查缓存再查 Serper
Firecrawl 500 正文抽取 官网公告、长文、详情页、结构化正文 只抽“已经被搜索命中且值得深抓”的页面
豆包 无限 主解析器 摘要、分类、实体归并、事件归因、结构化输出 日常任务主走豆包，把昂贵配额留给搜索与抽取
Tavily 1000 时效验证 突发争议、新闻型事件、交叉校验 只用于“需要确认是否真爆”的事件型问题
Exa 1000 语义扩展 长尾发现、语义搜索、冷门站点补足 不进默认热路径，只用于难搜或长尾问题
Gemini 按需及时使用 复杂 fallback 复杂页面理解、多模态内容解释、异常回退 不是默认主链，但可用于复杂 case 和补位

配额与调度

如果按月预算做折算，核心问题不是“总量够不够”，而是“哪些请求值得花配额”。下面这套规则会把有限额度集中用在真正能提升信号质量的地方。

Provider月预算日均安全线备注

Routing

默认路由规则

  • 发现层：`cache -> Serper`，只有 Serper 命中后才进入后续深抓。
  • 抽取层：若页面得分高、来源可信或内容长，再调用 `Firecrawl`。
  • 解析层：`豆包` 做摘要、实体归并、事件归因、置信度打分。
  • 时效校验：遇到争议、突发爆点、真伪不明，再触发 `Tavily`。
  • 长尾扩展：当 Serper 搜不到或命中质量差，再调 `Exa`。
  • 复杂 fallback：页面结构异常或需要更强理解时切 `Gemini`。
Hot Path

主链路推荐

发现候选
  -> 查 QueryCache
  -> Serper 搜索
  -> 本地去重与站点打分
  -> 命中高价值页面才触发 Firecrawl
  -> 豆包做结构化抽取
  -> Rust pipeline 清洗 / 去重 / 归并 / 事件归因
  -> 入库 + 星河页面刷新

精查候选
  -> Tavily 做事件验证
  -> Exa 做长尾补足
  -> Gemini 作为复杂 fallback

数据链路

最终页面中间是星系，底层系统也应该像星系一样层层向内收束。外层负责发现，中层负责抽取，核心层负责归并与生成信号。

Discovery Ring

外环：发现与召回

来源于搜索 API 和手工种子。职责是找到“可能值得看”的候选链接和结果，而不是把全部内容抓下来。

Extraction Ring

中环：正文抽取与标准化

Firecrawl 和自定义解析器负责抽正文，豆包负责把页面内容清洗成结构化输入。

Resolution Core

内核：Rust 归并中枢

统一进入 Rust pipeline，完成时间标准化、内容近重复、信息重复、实体归并、事件归因与信号落库。

Rust 架构

Rust 这一层不是“再包一层 API”，而是所有 provider 的统一调度枢纽。页面星系是产品表面，Rust scheduler 才是实际的动力核心。

Modules

建议模块拆分

  • providers/：Serper、Firecrawl、Exa、Tavily、豆包、Gemini 的 adapter
  • scheduler/：请求优先级、配额账本、回退策略
  • cache/：query cache、page cache、provider response cache
  • pipeline/：你现有的 clean / dedupe / resolution / event attribution
  • signals/：入库、快照刷新、前端读取接口
Serving

页面数据供给

  • 页面不直接依赖第三方 API，而是读 Rust 层统一产出的 signal 视图。
  • 热门赛道、事件时间线、来源结构、星系节点都从本地聚合表读取。
  • 这样前端只关心“怎么显示宇宙”，不关心“今天调用了哪个 provider”。

Provider 抽象

为了后面你给 API key 时可以快速接入，建议一开始就用统一 trait，把不同 provider 的能力做成“搜索 / 抽取 / 解析 / 校验”四类接口。

pub enum ProviderKind {
    Serper,
    Firecrawl,
    Doubao,
    Tavily,
    Exa,
    Gemini,
}

pub enum ProviderCapability {
    Search,
    Extract,
    Reason,
    Verify,
}

pub struct ProviderBudget {
    pub monthly_limit: Option,
    pub daily_soft_limit: Option,
    pub priority: u8,
}

pub trait SearchProvider {
    async fn search(&self, query: &str) -> anyhow::Result<Vec<SearchHit>>;
}

pub trait ExtractProvider {
    async fn extract(&self, url: &str) -> anyhow::Result<PageDoc>;
}

pub trait ReasonProvider {
    async fn structure(&self, input: &str) -> anyhow::Result<StructuredSignalDraft>;
}

Quota Ledger

真正决定这套系统能不能长期跑起来的，是配额账本和缓存，而不是 provider 数量。每次请求都要先问：这次值不值得花一个稀缺调用？

pub struct QuotaLedger {
    pub provider: ProviderKind,
    pub used_monthly: u32,
    pub used_today: u32,
    pub soft_limit_today: Option<u32>,
}

pub enum RouteDecision {
    UsePrimary,
    UseFallback,
    SkipAndCache,
    DelayForBatch,
}

// 示例策略
// 1. query 命中缓存 -> 直接返回
// 2. Serper 未超软上限 -> 继续搜索
// 3. Firecrawl 只给 score >= threshold 的页面
// 4. Tavily 只给争议 / 新闻型事件
// 5. Exa 只在搜索结果质量差时启用
// 6. 豆包默认常开，Gemini 只作复杂回退

落地阶段

为了不把设计、页面和抓取同时拉爆，建议按三段推进。第一段先把“星河骨架 + provider 抽象”立住，第二段再接真实调用，第三段才做强化视觉与实时联动。

阶段 A：页面框架与宇宙主视觉

先做新的前端壳：左导、星河 Hero、分析带、深度工作区。页面先吃 mock 数据，把空间感和信息层级定住。

阶段 B：Rust 调度层与真实 Provider 接入

实现统一 provider traits、quota ledger、query cache、page cache，并接你后续提供的 API key 和配置。

阶段 C：热点联动、实体下钻与风暴可视化

把赛道核心、事件风暴、来源结构变化和提醒环接成真正可交互的星图，而不是静态设计稿。

这份方案落地后的直接交付物

Frontend
