# 客户需求复用优先示例

这个示例用于说明：当用户有一个模糊需求时，这个 skill 应该先做什么。

它不是最终 prompt，也不是最终 `SKILL.md`。
它是一个“方案整理结果”的示例。

## 示例输入

`我想让 OpenClaw 帮我们团队处理客户需求，但我说不清到底是做一个 agent、一个 skill，还是直接找现成的东西来改。`

## 这个 Skill 应该先澄清什么

- 真正想解决的问题是什么
- 目标用户是谁
- 第一版什么样才算够用
- 有没有现成 skill 已经覆盖大部分需求
- 下一步更适合：
  - 直接复用
  - 轻改
  - 新做

## 一个好的结果应该长什么样

### 真实问题

- 团队在接收客户需求时，输入格式不一致，导致每次都要重新整理。
- 大量时间浪费在重复澄清和重复补问上。

### 期望结果

- 把原始客户需求整理成团队可以继续处理的结构化输入包。

### 复用优先判断

- 先扫描现有 intake、triage、request-normalization 类型的 skill。
- 如果有成熟 skill 已经覆盖：
  - 输入整理
  - 必要字段检查
  - 下一步建议
  就优先轻改，而不是从零开始。

### v1 范围

- 先只服务一个团队
- 先只处理一种需求类型
- 不接外部系统
- 不写代码

### 下一步

- 如果发现接近的现有 skill，就比较：
  - trigger
  - non-trigger
  - 输入边界
  - 输出边界
  - 依赖情况
- 如果没有合适的现成 skill，再进入新的 agent / skill 设计。
