# 客户需求分诊 Skill V1 交接示例

这个示例记录的是：从上游需求澄清结果，如何收敛出一个更窄、更稳的下游 skill 方向。

它不是最终 `SKILL.md`，而是一个正式写作前的交接结果。

## 核心问题

真正要解决的是：

- 把一段混乱的客户需求整理清楚
- 看清哪些信息已经明确
- 看清哪些信息还缺
- 看清风险、矛盾和歧义
- 产出一个可以继续往下处理的结构化结果

## 这个 Skill 是什么

`v1` 应该是：

- 一个客户需求分诊 skill
- 一个结构化整理与澄清 skill
- 一次输入，对应一次稳定输出

`v1` 不应该是：

- 销售助手
- 报价系统
- CRM 更新器
- 自动回复工具
- 建任务 agent

## 推荐的 v1 输入

- 客户文本
- 或者已经转写好的语音文本

`v1` 不依赖：

- 实时语音识别
- 外部系统
- 客户历史数据
- 价格逻辑
- 商务审批逻辑

## 推荐的 v1 输出

固定 6 栏：

1. 客户需求一句话版
2. 已知信息
3. 缺失信息
4. 风险或矛盾
5. 建议补问
6. 下一步分类

## 修正后的下一步分类

每次只给一个主分类：

1. `需要补充信息`
2. `可进入内部处理`
3. `存在风险需人工判断`

最多附一行简短理由。

不要输出：

- 报价
- 报价建议
- 商务结论
- 任务拆解
- 对客回复文案

## 为什么先定成这 3 类

这 3 类足够稳：

- `需要补充信息`：输入不完整时不假装能推进
- `可进入内部处理`：可以继续往下，但不假装已经开始执行
- `存在风险需人工判断`：避免 skill 对模糊或敏感内容做过度确定的判断

同时也能防止 skill 漂成：

- 销售判断器
- 流程编排器
- 执行批准器

## 明确的 v1 停止边界

输出结构化分诊结果后就停止。

不要：

- 回复客户
- 报价
- 建任务
- 调系统
- 做最终业务判断

## 正式写作时还要继续强调

当这个例子被写成正式 `SKILL.md` 时，要继续强调：

- 所有判断只基于当前输入
- `可进入内部处理` 不等于批准执行
- 这个 skill 只负责整理与澄清
- 最终怎么推进仍由人决定
