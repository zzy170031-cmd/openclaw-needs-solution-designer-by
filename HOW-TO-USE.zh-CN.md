# 我自己应该怎么用

这份文档是给你自己和以后使用这个仓库的人看的。

它的目标很简单：

**告诉你怎么真正把这套 skill 用起来，而不是只看懂它的理念。**

## 这个 Skill 最适合什么时候用

当你出现下面这些情况时，就适合用它：

- 我有一个想法，但现在还说不清楚
- 我知道有问题，但不知道最适合 OpenClaw 的路径是什么
- 我不知道应该直接复用、轻改，还是新做
- 我想先拿到一份清晰方案文档，再继续正式写作或执行

最重要的一点：

**不要一开始就逼自己用技术术语。**
白话输入才是这套 skill 的正常入口。

## 最简单的开始方式

在 Codex 里新开一个线程，然后直接说：

```text
Use $openclaw-needs-solution-designer-by to help me.

我有一个想法，但现在说不清楚。
```

或者更直接一点，就把你的白话想法丢进去：

```text
Use $openclaw-needs-solution-designer-by to help me.

我想让你帮我推送信息。
```

## 轮询时你应该怎么回答

你不需要一次说得很完整。
短句、直说、只回答关键点，就够了。

好的回答方式：

- “给销售团队用。”
- “我只关心 AI 新闻。”
- “发到飞书。”
- “第一版先做小一点。”

如果它已经复述得差不多对了，就明确确认。

好的确认方式：

- “对，就是这个意思。”
- “没错，我就是想这样。”

如果它理解错了，不要整段重写。
直接指出哪一句不对就行。

好的纠正方式：

- “不是给客户，是给内部同事用。”
- “不是实时提醒，我要的是日报。”

## 用得好时，你最终应该拿到什么

这套 skill 正常工作时，最后应该帮你拿到：

- 更清楚的问题定义
- 更清楚的目标结果
- 一个 `复用 / 轻改 / 新做` 的判断
- 一个稳定的 v1 范围
- 明确的 blockers 和 assumptions
- 一份 OpenClaw 执行指引

## 最推荐的使用流程

最健康的用法是：

1. 先用白话开头
2. 让它通过多轮轮询帮你讲清楚
3. 当它总结得差不多对时，明确确认
4. 让它继续把 v1 范围收小
5. 最后让它输出方案文档和 OpenClaw 执行指引

## 怎么判断现在是不是用对了

好的迹象：

- 它一直在用大白话
- 它能区分 confirmed 和 assumptions
- 它问的是最少但关键的问题
- 它不会太早跳进代码
- 它会帮你看清楚哪些东西现在先不要做

不好的迹象：

- 它开始脑补很多你没确认过的细节
- 它太早把你拉进 agent / skill 术语
- 需求还没稳定，就开始写代码、写 prompt、写最终 `SKILL.md`

## 拿到结果以后你怎么继续

当它已经给你一份稳定结果后，下一步一般有 3 种：

- 直接继续交给 OpenClaw
- 如果确定要新建 skill 或 agent，再把结果交给设计器类 skill
- 如果暂时不需要开发，就把它当成更清晰的流程文档保存下来

## 三种常用开场

```text
我想让 OpenClaw 帮我做点事，但我现在还说不清楚。
```

```text
我怀疑这个需求网上已经有类似方案了，先帮我判断值不值得从零做。
```

```text
我已经有一版草稿了，但我想先把问题、范围和下一步整理清楚。
```
