# 更新路线图

这份文档描述 `openclaw-needs-solution-designer-by` 在首发后的演进方向。

整个路线图有一个最高原则：

**不要为了看起来更快，而牺牲结果的稳健性。**

这个 skill 仍然要坚持它最初的使命：
帮助不懂代码的用户，把模糊需求整理成清晰方案文档和可继续交给 OpenClaw 的执行指引。

## 产品底线

后续所有更新都应该保留这些基础：

- 白话输入仍然是默认入口
- 多轮轮询澄清仍然是产品机制
- `confirmed facts` 和 `working assumptions` 必须分开
- `reuse / adapt / build new` 的判断必须明确
- 最终输出必须仍然能真正帮助用户继续使用 OpenClaw

## v1.0.x：先把首发版跑稳

目标：

- 验证首发版在真实使用中是否稳定

重点：

- 收集 5 到 10 条真实使用对话
- 观察轮询轮次是太多还是太少
- 继续优化“确认提示语”
- 继续优化 `Working assumptions`
- 优化最终的 OpenClaw 执行包格式

成功信号：

- 用户能稳定得到一份清晰方案，而不会被拖进不必要的技术表达里

## v1.1：把复用判断做得更稳

目标：

- 让 `直接用 / 轻改 / 新做` 的判断更可靠

重点：

- 增强现有 skill 对比示例
- 把对比维度写得更清楚
- 降低“只因为看起来像就被误判为可复用”的情况
- 提升“不适合复用时”的解释质量

成功信号：

- skill 能用非技术用户听得懂的大白话解释：为什么应该复用，或者为什么不该复用

## v1.2：把白话入口继续做强

目标：

- 更擅长接住高度模糊、很口语化的需求

重点：

- 扩充 `samples.md` 里的白话入口例子
- 优化任务类型归一化
- 优化前 1 到 2 轮的最小澄清
- 让用户更容易意识到“现在轮到确认了”

成功信号：

- 用户只说一句很模糊的话，也能被稳稳带进清晰方案，而不是被技术术语吓住

## v1.3：把最终输出包做得更能直接落地

目标：

- 让最终结果更适合继续交给 OpenClaw 使用

重点：

- 强化方案文档结构
- 强化 OpenClaw 执行指引格式
- 增加更多“可直接继续执行”的示例
- 更明确地告诉用户：哪些事现在还不该让 OpenClaw 做

成功信号：

- 用户拿到输出后，不需要自己再翻译一遍，就知道下一步怎么继续交给 OpenClaw

## v1.4：补真实案例库

目标：

- 让新用户更容易看懂怎么用这套 skill

重点：

- 增加不同类型的真实案例
- 增加 before / after 示例
- 增加这些类型的例子：
  - 直接复用
  - 轻改
  - 新做
  - 保持为更简单的流程/模板

成功信号：

- 新用户不只看原则，也能通过案例快速理解这套 skill 的工作方式

## v2.0：可选增强，而不是核心膨胀

目标：

- 支持更多场景，但不把核心 skill 做成大杂烩

可能加入：

- 可选行业案例包
- 可选常见场景模板
- 可选更强的执行包参考资料

明确不追求：

- 把核心 skill 做成一个巨大的行业百科
- 用漂亮文案掩盖模糊假设
- 用自动化替代用户确认

## 发布后的实际建议

最值得做的动作是：

1. 持续保存真实使用对话
2. 观察用户在哪些地方会犹豫或误解
3. 只补最小必要规则，不盲目扩架构
4. 先长案例，再长机制
